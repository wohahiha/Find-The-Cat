## 完成的功能

### ✅ P0 阶段：基础架构

#### 1. 日志标准文档（`docs/日志标准.md`）
- 定义了FTC日志的4个级别：INFO, WARNING, ERROR, CRITICAL（不使用DEBUG）
- 规定了2种输出格式：PLAIN（人类可读）和 JSON（机器可读）
- 定义了9个核心字段：timestamp, level, logger, message, username, account_id, user_id, ip_address, request_path
- 制定了日志分类标准：业务日志、框架日志、系统日志、其他

#### 2. 格式转换模块（`apps/system/log_formatters.py`）
- `to_plain()`: 将LogEntry转换为PLAIN格式字符串
- `to_json()`: 将LogEntry转换为JSON格式字符串
- 支持双向转换和格式验证

#### 3. 日志解析类（`apps/system/log_models.py`）
- `LogEntry`: 日志条目数据类，包含9个核心字段 + 3个计算方法
  - `get_type()`: 自动分类为业务/框架/系统/其他日志
  - `get_source()`: 提取日志来源模块名
  - `get_summary()`: 智能截断消息摘要
- `LogParser`: 日志文件解析器，支持PLAIN和JSON两种格式
  - 使用生成器模式，内存友好
  - 支持limit参数控制读取数量
  - 自动检测格式并解析

#### 4. 系统配置调整（`apps/system/services.py`）
- 新增配置项：`LOG_OUTPUT_LEVEL`（3个级别：Low/Medium/High）
  - Low: 仅输出ERROR + CRITICAL
  - Medium: 输出WARNING + ERROR + CRITICAL（默认）
  - High: 输出INFO + WARNING + ERROR + CRITICAL
- 新增配置项：`LOG_PATH`（日志文件路径，默认 logs/）
- 修改日志文件名：从 `ftc.log` 改为 `system.log`

#### 5. Account ID 功能增强
- 新增异常：`AccountIdLimitError`（apps/common/exceptions.py）
- User模型新增字段：`account_id`（唯一、自增、不可为空）
- 实现分段分配策略：
  - 超级管理员：1-10
  - 管理员：11-1000
  - 普通用户：1001+
- 创建数据迁移为现有用户分配account_id
- Admin界面显示account_id字段
- 限制超级管理员数量上限为10个，仅能通过命令行创建

### ✅ P1 阶段：日志记录器重构

#### 1. LOG_OUTPUT_LEVEL级别映射（`apps/common/infra/logger.py`）
- `get_log_level_from_config()`: 从SystemConfig读取配置并映射到Python logging级别
  - Low → logging.ERROR (40)
  - Medium → logging.WARNING (30)
  - High → logging.INFO (20)

#### 2. 符合标准的格式化器
- `FTCJSONFormatter`: JSON格式化器
  - 输出9个字段的JSON对象
  - 自动注入请求上下文
  - 使用`ensure_ascii=False`支持中文
- `FTCPlainFormatter`: PLAIN格式化器
  - 格式：`时间戳 级别 logger 消息 [账户名|账户ID|IP|路径]`
  - 示例：`2025-11-29 05:32:36 WARNING __main__ 测试日志 [admin|1|127.0.0.1|/test/api]`

#### 3. 日志轮转配置
- 使用`TimedRotatingFileHandler`
- 每日午夜自动轮转
- 保留30天历史日志
- 日志文件名格式：`system.log`, `system.log.2025-11-28`, ...

#### 4. 上下文自动注入
- 更新`apps/common/utils/request_context.py`：新增`account_id_ctx`变量
- 更新`apps/common/middleware.py`：从User对象提取account_id并注入上下文
- 所有日志自动包含：username, account_id, user_id, ip_address, request_path

#### 5. AdminAuditMixin重构（`apps/accounts/admin.py`）
- 简化代码，移除手动extra参数传递
- 依赖自动上下文注入
- 符合新日志标准

#### 6. Django设置更新（`Config/settings.py`）
- 禁用Django默认LOGGING配置：`LOGGING_CONFIG = None`
- 添加废弃配置说明注释

#### 7. App初始化（`apps/system/apps.py`）
- 在`ready()`方法中调用`configure_logging()`
- 确保Django启动时自动初始化日志系统

### ✅ P3 阶段：4个过滤器

#### 1. 级别过滤器（LevelFilter）
- 可选值：INFO, WARNING, ERROR, CRITICAL

#### 2. 类型过滤器（TypeFilter）
- 可选值：业务日志、框架日志、系统日志、其他

#### 3. 来源过滤器（SourceFilter）

#### 4. 账户类型过滤器（AccountTypeFilter）
- 可选值：

  - 管理员 (ID 11-1000)
  - 普通用户 (ID 1001+)

## 使用指南

### 1. 后台查看日志

```bash
# 启动开发服务器
python manage.py runserver

# 访问Admin后台
http://localhost:8000/admin/

# 导航路径
SYSTEM → 系统日志分类 → 系统运行日志
```

### 2. 使用过滤器

在日志列表页面右侧，可以看到4个过滤器：
- **日志级别**: 选择INFO/WARNING/ERROR/CRITICAL
- **日志类型**: 选择业务日志/框架日志/系统日志/其他
- **日志来源**: 选择模块名
- **账户类型**: 选择超管/管理员/用户/未登录

### 3. 导出日志

访问以下URL直接下载对应格式的日志：
```
http://localhost:8000/admin/configs/systemlog/export/txt/
http://localhost:8000/admin/configs/systemlog/export/csv/
http://localhost:8000/admin/configs/systemlog/export/json/
```

### 4. 配置日志级别

```bash
# 进入Admin后台
SYSTEM → 基础配置

# 找到 LOG_OUTPUT_LEVEL 配置项
# 可选值：
# - Low: 仅输出错误（ERROR + CRITICAL）
# - Medium: 包含警告（WARNING + ERROR + CRITICAL）- 推荐生产环境
# - High: 全部日志（INFO + WARNING + ERROR + CRITICAL）- 推荐开发环境

# 修改后需重启Django服务生效
```

### 5. 在代码中记录日志

```python
from apps.common.infra.logger import get_logger

logger = get_logger(__name__)

# 自动包含上下文（username, account_id, ip等）
logger.info("用户登录成功")
logger.warning("密码即将过期")
logger.error("支付失败")
logger.critical("数据库连接丢失")
```

## 注意事项

### 1. 数据库初始化警告
启动时可能看到以下警告（正常现象）：
```
RuntimeWarning: Accessing the database during app initialization is discouraged.
```
这是因为日志系统在`apps.ready()`中需要读取SystemConfig，属于预期行为。

### 2. 日志文件轮转
- 系统会自动在每日午夜轮转日志文件
- 旧日志保留30天后自动删除
- 确保`logs/`目录有写入权限

### 3. 性能考虑
- Admin日志展示最多读取1000条
- 导出功能最多导出10000条
- 如需查看更多历史日志，请直接访问日志文件

### 4. 超级管理员限制
- 系统限制最多10个超级管理员
- 仅能通过命令行创建：`python manage.py createsuperuser`
- Admin界面的"添加管理员"功能不能创建超级管理员
