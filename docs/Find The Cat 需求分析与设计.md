# CTF竞赛平台-Find The Cat需求分析与设计

## 1 项目背景与目标

随着网络安全攻防竞赛（CTF）的发展，各类开源 CTF 平台纷纷涌现，包括 CTFd、FBCTF、RootTheBox、rCTF、Mellivora 等。这些平台为比赛主办方提供了题目发布、计分排行等基础功能，但各有侧重与不足：CTFd 易于使用且支持插件和主题，但对动态靶场支持有限；RootTheBox 功能丰富（甚至加入 Botnet 等游戏元素），但部署相对复杂；FBCTF 具有独特的地图界面和攻防模式，但适用场景有限；rCTF 注重现代化架构和性能扩展，却需要配套独立的题目部署平台。在高校网安协会或战队自主举办 CTF 赛事时，希望有一个功能全面强大的竞赛平台，既能整合上述 CTF 平台的优点，又不直接依赖某单一项目，以保持自主性和高扩展性。

因此我创建了本项目：Find The Cat，简称 FTC，旨在设计并实现一套功能丰富、扩展性强的 CTF 竞赛平台。FTC 采用前后端分离架构（Django + Vue.js），以部署便利和使用顺畅为核心设计原则，覆盖从赛前题目部署、赛中竞赛管理到赛后复盘的全流程功能。FTC 融合现有开源 CTF 框架的优点，取长补短，创建新一代的 CTF 竞赛平台。FTC 全面支持 CTF 竞赛各环节，包括题目管理、参赛答题、计分排名、团队管理等，并注重系统的安全性和扩展性，以适应不同规模比赛需求和未来功能拓展。

## 2 设计目标

构建一个免费开源、易于维护的 CTF 平台，满足如下目标：

- **全面功能**：FTC 涵盖 CTF 竞赛的常见需求（题目、用户、队伍、计分等），并引入业界优秀 CTF 平台/框架的特色功能，确保竞赛功能强大且灵活。FTC 支持题目导入导出、动态靶机环境、团队赛等全面的功能点，满足从日常训练赛到大型线上赛的各种需求。
- **高可用性和性能**：FTC 支持大规模赛事的高并发，保证平台响应迅速。通过数据库优化、缓存等手段提升性能，在千人以上规模赛事时依然能够确保页面加载和判题反馈的及时。
- **轻量部署**：FTC 架构简洁，依赖组件全部为免费开源软件，支持在低配置服务器上平稳运行，并可通过 Docker 等手段实现快速部署。
- **安全可靠**：FTC 自身具备良好的安全防护，防止常见 Web 安全漏洞（SQL 注入、XSS、CSRF 等），并对竞赛过程中可能出现的恶意行为有防范措施，保证竞赛数据的完整性和隔离性。
- **易扩展**：FTC 采用模块化设计，并预留扩展接口，方便后续增加新功能或对接第三方服务（如 Attack-Defense 攻防赛制等）。FTC 通过插件机制和松耦合架构实现灵活的可扩展性，能够添加新的题目类型、比赛模式或集成外部工具，而无需改动核心系统。

## 3 功能需求分析

### 3.1 用户及角色管理

- **用户注册登录**：FTC 提供用户注册、账户登录和密码找回功能。用户注册时需填写用户名、邮箱和密码，并通过邮箱完成验证。注册成功后，用户即可使用用户名或邮箱 + 密码登录 FTC。登录成功后 FTC 通过 JWT（JSON Web Token）机制维持用户的会话状态，系统会签发加密的 Token 令牌，前端在访问受限 API 时需附带该 Token 以验证身份。若用户忘记密码，FTC 支持通过邮箱验证的方式找回密码。此外，为抵御恶意批量请求，FTC 在注册、登录和找回密码等流程中加入了轻量级的人机验证机制。该机制基于开源项目 [CAP](https://capjs.js.org/) 引入工作量证明（Proof-of-Work）挑战，以确保只有通过验证的请求才能继续操作。FTC 预留了手机号登录和第三方登录的接口，以支持日后集成如手机号、GitHub、微信等账户登录。
- **角色权限控制**：FTC 主要包含选手和管理员两类角色。选手（包括队伍成员）登录后只能访问比赛答题相关功能，如浏览题目、提交 Flag、查看自身成绩等；管理员则拥有后台管理权限，能够管理题目、比赛、用户等。FTC 也支持角色细分，例如队伍中选手可分为队长和队员，管理员可分为出题人、裁判等，系统会根据比赛需求赋予不同角色以不同的权限。
- **个人资料管理**：用户可以查看和更新个人资料，包括昵称、邮箱、密码等基本信息（用户名一经创建，只可注销，不可修改）。其他非必要信息（例如性别、头像等）不作强制要求，用户可选择不填写或自行设置。

### 3.2 比赛与团队管理

- **比赛配置**：管理员可以通过后台创建新的比赛，并配置比赛信息，如比赛名称、开始和结束时间、比赛说明等。支持将比赛设置为公开赛或私有赛（私有赛仅对受邀用户开放）。比赛可开启排行榜“封榜”功能，即在比赛最后一段时间冻结排行榜显示（详见3.5-排行榜展示）。比赛开始后，参赛选手可以提交 Flag 获取分数；比赛结束时间一到，FTC 自动停止计分，超出时间的提交将不再计入成绩。
- **团队参赛**：FTC 支持团队赛模式。用户可以自由创建队伍，设置队伍信息并生成队伍加入令牌，其他选手通过该令牌加入团队。每位用户在同一场比赛中最多只能加入一支队伍；加入团队后，个人的有效提交将计入团队成绩。团队成绩按照团队成员的解题情况汇总计算，队伍成员可以在排行榜中查看所在团队的总分排名。
- **公告通知**：提供比赛公告栏功能，管理员可以在比赛过程中发布公告或更新（例如题目勘误、规则变更等）通知所有选手。选手登录后会在显著位置看到最新公告，确保及时知晓重要信息。对于紧急或重要公告，FTC 还会以弹窗方式高亮提示。公告功能支持纯文本和富文本（Markdown）两种格式，以便清晰准确地传达信息。

### 3.3 题目管理与题库功能

- **题目录入与编辑**：管理员可通过后台界面添加和编辑赛题，录入内容包括题目名称、描述、类别、难度、初始分值等字段。题目描述支持纯文本和 Markdown 富文本，以便展示代码、图片等内容。每个题目可设置多种标签（如 Web、Pwn、Misc 等类别标签及难度标签），实现分类管理。FTC 支持上传附件文件（如源码、图片、流量包等）供选手下载，并允许附加题目链接或提示信息。题目编辑功能支持版本更新，管理员可在发现错误时修改题目信息并通知选手。
- **Flag 与校验**：每个题目需配置 Flag 及其验证方式。FTC 支持 静态 Flag 与 动态 Flag 两种类型。静态 Flag 由管理员直接填写，系统在后端比对判定；动态 Flag 由系统在选手启动靶机时自动生成，管理员提供生成模板或参数，平台为每位选手创建独立 Flag 并注入容器环境。判题时系统统一通过后台接口校验提交结果，确保不同类型题目均可通过同一提交流程完成验证。
- **题目库与复用**：FTC 引入“题库中心”概念，将题目数据与具体比赛分离。已录入的题目作为题库资源，可以在新的比赛中直接引用，实现“一题多赛”。FTC 还支持将题目以文件或数据库形式导出备份，并在其他比赛中导入复用。通过题库，CTF 主办方能够逐步积累自己的题目资产，提高出题效率。需要注意的是，题库中题目的变更不会影响已结束比赛的数据，即题库相当于比赛原题目的一个拷贝。
- **多子任务题目**：FTC 支持一道题目包含多个子题或多个 Flag（Multi-Flag）。一个大型综合题可以由若干子挑战组成，管理员在录入题目时可以将题目拆分为多个子挑战，并为每个子挑战配置对应的 Flag 和分值。选手每解出一个 Flag 即可获得对应分值，FTC 会在选手提交 Flag 时自动判断其所属的子挑战并记录得分。多 Flag 的设计增加了题目的灵活性，鼓励选手从多个方面探索题目。
- **附件与环境配置**：许多题目需要附件或在线运行环境。FTC 提供统一的附件上传与下载服务，并支持为题目关联靶机环境。出题者可在配置中指定题目的 Docker 镜像或 Docker Compose 模板，由系统自动部署运行容器（详见“靶机管理”部分）。同时，管理员可为题目设定资源限制（如容器内存、CPU、端口范围等），确保不同题目环境相互隔离、资源分配合理。

### 3.4 靶机管理与动态环境支持

- **Docker 靶机实例**：为保证每位参赛选手在解题时拥有相互隔离、不受干扰的环境，FTC 支持私有化的靶机部署方式。对于需要在线运行服务的题目（如部分 Web 题、Pwn 题），每个选手可以通过平台请求启动一个专属该题目的 Docker 容器实例。FTC 将从预先准备的镜像启动容器，并随机映射容器内部服务端口到平台服务器的空闲端口供该选手使用。管理员可为每道题设置容器实例的并发上限（即同时运行的实例数量上限），以防止资源耗尽或性能下降。当达到上限时，后续启动请求会被拒绝，并提示选手关闭空闲实例后再尝试。
- **容器生命周期管理**：FTC 自动维护靶机容器的生命周期。当选手在前端点击“启动靶机”按钮时，后台记录一个启动请求并调用 Docker API 运行该题目的容器。容器启动后，FTC 允许其运行一段预设的最长存活时间（例如默认 1 小时），以防止容器长时间占用资源。FTC 后台会定期检查已经到期的容器；当容器超过设定存活时间或比赛结束后，系统将自动销毁该容器释放资源。当然，选手也可以通过前端界面手动停止并释放靶机。
- **后端扩展预留**：FTC 支持通过 Docker 镜像或 Docker Compose 创建容器实例，并在架构上预留了对其他类型靶机环境的扩展接口，便于未来集成如虚拟机、Docker Swarm、Kubernetes 集群或基于 VPN 网络隔离的靶场体系。通过统一的靶机控制与调度接口，FTC 能够在不改变核心逻辑的前提下平滑接入多种异构环境，为 Attack-Defense 等复杂赛制提供扩展空间。
- **靶机访问与端口管理**：FTC 采用平台服务器统一转发流量的方式处理靶机通信，无需人工开放大量端口。当容器启动后，平台记录容器内部服务端口，并在服务器上开放对应的映射端口，再将选手对该端口的访问流量转发至靶机容器。管理员只需提前在服务器防火墙开放一小段端口范围，FTC 即可动态分配和管理端口，而无需为每个容器手动设置映射，从而大幅降低了部署难度。
- **Checker 脚本（靶机侧校验）**：默认情况下，题目的 Flag 校验在平台后端完成。然而，FTC 同时支持靶机容器内部校验模式。管理员可以在靶机环境中包含一个 checker 校验脚本，选手提交 Flag 后，FTC 将调用该脚本在容器内进行校验。FTC 默认采用平台后端校验 Flag，管理员可根据需要启用靶机内 checker 脚本选项，并配置相应的校验脚本路径。注意，Flag校验方式与Flag类型（静态/动态）无直接联系。

### 3.5 提交答题与计分系统

- **提交控制**：为防止选手通过脚本暴力猜解 Flag，FTC 对 Flag 提交行为进行了严格控制。系统默认限制每位选手的提交频率，管理员可以根据实际需要调整这一限制参数。FTC 还会为所有提交生成日志，包含用户、内容、时间和结果等信息，管理员可通过后台查看日志以检查选手是否存在异常提交行为。对于存在恶意提交嫌疑的用户，FTC 将自动给予警告并在后台提示管理员注意。
- **计分规则**：FTC 支持静态计分和动态计分两种模式。静态计分模式下题目分值固定不变；动态计分模式下题目分值会随解出人数逐渐下降，以奖励率先解题的选手。不论个人赛或团队赛，每个参赛者或队伍对同一道题目只计分一次；团队赛中仅队内首位解出该题的成员为团队贡献得分，其他成员后续的正确提交被视为重复且不计分。动态计分题目的具体变动规则由管理员自定义，FTC 在选手提交正确 Flag 时依据预设规则计算其得分。
- **排行榜展示**：FTC 提供实时更新的排行榜页面，按选手或队伍显示总分排名。系统支持按题目标签（Tags）筛选生成不同维度的排行榜（例如 “Pwn 排行榜”、“简单题排行榜”、“学校名排行榜”等），以多维方式展现选手表现。FTC 还支持“封榜”（Freeze Scoreboard）功能：在比赛最后一段时间停止向选手端更新排行榜，但后台仍记录实际提交情况，待比赛正式结束后解除封榜并公布最终结果。
- **“n血”奖励**：FTC 支持为每道题的前 n 位（默认前3名）解题选手给予额外奖励。例如管理员可以给予前三名解出题目的队伍额外的分数奖励，又或是在动态分值的题目中采取前“三血”分值不变的机制，以激励选手抢先答题。FTC 还会在排行榜上显示每位选手（或队伍）的 “n血” 数量，以进一步展现参赛者的解题速度和实力。
- **提示与罚分**：FTC 支持题目提示功能，管理员可为每道题配置若干提示信息。提示分为两类：免费提示和扣分提示。选手查看免费提示不影响题目分值，而查看扣分提示则需要选手确认扣除相应分值后才能查看。通过 Hint 机制，选手在卡关时可以以扣减分数为代价获取提示；管理员可以借此平衡题目难度，避免出现无人解出的情况。

### 3.6 其他功能性需求

- **日志与监控**：FTC 会记录关键操作日志，包括用户登录日志、Flag 提交日志、管理员操作日志等，便于安全审计和赛后分析。管理员后台提供基本的监控界面，可以查看当前在线人数、靶机运行数量、系统资源占用等系统状态。
- **数据导出与备份**：比赛结束后，管理员可以导出比赛数据（包括题目列表、用户成绩等）用于成绩统计分析或备份归档。比赛期间 FTC 也提供数据库定期备份机制，以保证数据安全。

### 3.7 用例场景示意

下面通过一个基础的用例场景，说明 FTC 的使用流程：

- **比赛设置**：竞赛管理员创建一场新的 CTF 比赛，设置比赛名称、时间、赛制（个人赛/团队赛）等参数，并通过后台批量导入赛题（包括题目描述、附件以及关联的 Docker 靶机镜像配置等）。管理员将比赛设置为公开比赛，并开启了排行榜封榜功能（例如比赛最后一小时封榜）。
- **用户报名**：比赛开始前，选手在 FTC 上注册账户并登录。选手可以自行创建或加入一个战队：例如选手 A 创建队伍 “AlphaTeam”，成为队长，系统生成加入令牌；选手 B 输入该令牌加入 “AlphaTeam”，成为队员。
- **解题过程**：比赛开始后，选手登录 FTC 查看题目列表。选手 A 打开一道 Web 类题目，阅读题目描述，发现该题需要在线靶机环境，便点击“启动靶机”，FTC 后台为他启动一个专属 Docker 容器，并返回连接信息（URL）。选手 A 使用提供的URL访问靶机进行开始解题，拿到 Flag 后在 FTC 提交答案。与此同时，选手 B 则专注于解答一道Misc题目，他直接下载题目附件，在本地分析破解得到 Flag 并提交答案。
- **答案提交与反馈**：每当选手提交 Flag，FTC 会立即校验其正确性。对于正确的 Flag，系统返回“解题成功”的提示，并记录该队伍解出此题，同时更新团队的总分和排行榜名次；对于错误的提交，FTC 则给出错误提示，并记录失败尝试次数。如果有选手成为某题的第一解出者，FTC 会在排行榜上对其进行 “First Blood” 标记并给予额外的奖励分（如果比赛规则设定了额外加分）。
- **通知与监控**：比赛进行中，管理员发现某题的题面描述存在歧义，于是更改该题藐视并在公告板发布澄清说明，所有在线选手都即时收到了弹窗公告提示。管理员通过后台监控页面查看当前的在线人数和每道题目的靶机容器运行情况。当发现某个队伍提交 Flag 的次数异常频繁时，管理员怀疑其可能在进行暴力猜解行为，便通过日志审计保留了相关证据。

## 4 非功能性需求

- **性能与并发**：FTC 能够支持上千名选手同时在线竞赛，并保证页面加载和提交判题的响应及时。系统通过数据库优化和引入缓存机制等手段提升性能，例如对排行榜等高频查询的数据使用 Redis 进行缓存（在大规模赛事中定时刷新缓存以减轻数据库压力）。同时还考虑了服务器在并发运行大量 Docker 靶机容器时的资源开销，确保在容器大规模启动的情况下依然维持平台的稳定运行。
- **安全与隐私**：FTC 从多个层面保证系统的安全与用户隐私。所有数据传输都经过 HTTPS 加密，防止信息被窃取。系统使用 JWT 令牌 进行身份验证，确保只有登录用户才能访问受限接口。注册、登录和找回密码等功能都加入了 CAP（轻量级人机验证），防止机器人恶意请求。数据库中保存的密码经过加密处理，不会存储明文。平台还对频繁的提交或访问行为设置了限速机制，防止暴力破解。管理员的关键操作（例如删除题目、修改分数等）都会被记录在安全日志中，以便赛后追踪和审计。
- **可用性与稳定性**：FTC 需要在比赛期间长时间保持稳定运行。系统采用模块化架构，各部分可以独立运行，即使某个服务出现问题，也不会影响整个系统。平台支持多实例部署和自动重启机制，保证服务在异常情况下能迅速恢复。对于高并发访问，FTC 通过缓存和数据库优化来保持良好的响应速度。排行榜、题目等高频数据会被缓存到 Redis 中，当数据库压力过大时，系统也能保持平稳运行。即使在大型比赛中，FTC 也能确保选手提交和页面访问不受干扰。
- **可扩展性与扩容能力**：FTC 在设计时充分考虑了未来的扩展需求。后台服务是无状态的，可以方便地在多台服务器上同时运行，从而应对更多用户并发。Redis 缓存和数据库可以通过主从复制或集群模式提升性能。当比赛规模扩大时，可以轻松增加靶机服务器节点，FTC 的靶机管理模块会自动分配任务到不同节点上。系统还预留了与 Kubernetes、虚拟机等工具对接的接口，未来若需部署更大规模赛事，只需连接这些接口即可扩展，而不需要修改核心代码。
- **易用性与用户体验**：FTC 的界面简洁直观，比赛相关的信息（公告、倒计时、分数变化等）都能在显眼位置实时显示。系统支持深色和浅色主题切换，并自适应电脑和手机端，方便选手随时参赛。输入表单和错误提示信息清晰明确，帮助用户快速定位问题。对于管理员，后台界面布局合理、分类清楚，可以方便地进行题目管理、公告发布、靶机监控等操作，让平台的使用和维护都更加高效。

## 5 FTC架构设计

### 5.1 技术栈选型

核心技术栈及选型理由如下：

- **后端（Django）**：Django 功能完备且稳定，自带 ORM、权限管理等丰富特性，适合开发复杂 Web 应用。Python 语言简单高效，并拥有丰富的安全和数据库类库，有利于实现 CTF 平台的一些特殊逻辑（如动态 Flag 校验脚本）。Django 开源免费且社区活跃，维护者容易上手。选择 Django 也符合团队的技术背景，并可结合 Django REST Framework 提供 REST API 接口，实现前后端分离架构。
- **前端（Vue.js）**：Vue 是现代主流前端框架，采用组件化开发提高了前端交互体验，适合构建友好的单页应用界面。Vue 的学习曲线平缓且有大量中文社区支持，非常适合本项目的前端开发需求。借助 Vue，我们实现了前后端解耦的 B/S 架构，由前端负责页面呈现与交互，后端提供纯粹的 JSON 数据接口。这种架构能够提高开发效率和用户体验，使前端界面更加动态流畅。
- **数据库（MySQL）**：MySQL 社区版作为免费的关系型数据库，性能可靠，可满足 CTF 平台读多写少的典型负载需求。其 ACID 特性保障了比赛数据（提交记录、计分等）的正确性。对于几百并发用户规模的比赛，单机 MySQL 完全可以支撑，并可通过合理的索引和优化获得快速查询性能。此外 MySQL 运维生态成熟，后续扩展（如主从复制、高可用方案等）有丰富经验可借鉴。选择 MySQL 也方便与 Django ORM 的整合，并避免商业数据库的授权成本。
- **缓存（Redis）**：采用 Redis 作为缓存数据库，用于存储热点数据（如排行榜、会话信息等）以加速读写响应。Redis 是开源的内存数据库，读写性能极高，用作缓存可有效减轻主数据库压力。例如，当参赛人数较多时，FTC会将排行榜数据定期写入 Redis，让前端从缓存获取结果，从而避免频繁的实时计算查询。Redis 还用作 Django 会话和 Celery 消息队列的后端，使 FTC 的整体结构更为简洁统一。
- **任务队列（Celery）**：FTC 的核心交互操作大多可同步完成，原则上不强依赖独立的消息队列中间件，从而降低架构复杂度。不过为了提高扩展能力，FTC预留了 Celery + Redis 作为可选的异步任务方案，可用于发送通知邮件、批量导出成绩等非实时任务。Celery 在 Python 生态中有良好的兼容性，结合 Redis 作为 Broker 能进一步地提升FTC的可用性。
- **容器化（Docker）**：FTC 采用 Docker 运行题目靶机环境，后端通过 Docker API 管理容器的创建、启动、停止与销毁，并记录端口和状态。系统支持两种创建方式：Docker 镜像和 Docker Compose 模板，平台自动解析题目配置并管理容器生命周期。FTC默认在一台物理服务器上运行 Docker 引擎，但靶机管理进程与主服务解耦，以便按需扩展至多节点。Kubernetes和 Docker Swarm的接口已在调度层预留，未来可直接接入 K8s 控制面以支持更大规模编排，无需修改上层逻辑。
- **Web 服务器（Nginx）**：FTC 前端采用开源且高性能的 Nginx 作为 Web 服务器和反向代理。Nginx 用于托管前端静态资源（即打包后的 Vue 文件）并将 API 请求转发给 Django 后端，从而提高并发处理能力。同时 Nginx 上配置 HTTPS 十分方便，有利于实现安全的访问。FTC 后端的部署采用 Gunicorn 搭配 Nginx 的经典方案，以获得稳定高效的生产环境性能。
- **开发与工具**：开发环境方面，FTC 使用 Python 虚拟环境来管理后端依赖；前端使用 Node.js（npm）进行构建打包。代码版本管理采用 Git。测试方面，后端利用 Django 自带的测试框架进行单元测试，前端则使用 Jest 等工具进行组件测试，从而保障 FTC 功能的正确性和稳定性。

### 5.2 FTC架构概览

FTC 采用典型的分层架构，前后端分离部署。主要的系统构件及它们之间的交互关系包括：浏览器前端（选手端、管理员端）通过 HTTP(S) 请求访问前端应用，前端再通过 API 与后端服务进行数据通信；后端连接数据库和缓存服务以存储和获取数据；对于需要交互环境的题目，后端还会与 Docker 容器交互（用于启动/销毁靶机实例），并通过服务器开放端口转发，让选手能够访问各自对应的靶机服务实例。

从功能上来说，FTC 划分为以下三个主要子系统：

- **用户与团队子系统**：负责用户注册登录、身份认证、权限验证、团队创建加入等，与账户和身份管理相关的功能模块。
- **题目与计分子系统**：涵盖题目管理与展示、Flag 校验、分值计算、排行榜维护等核心竞赛逻辑。（该模块与数据库交互频繁，包含动态计分、多任务题等复杂业务规则）
- **靶机管理子系统**：处理动态靶机环境的分配与控制，包括 Docker 容器的启动、资源限制分配以及销毁调度等。（该模块相对独立，可作为后台服务进程部署，在需要时扩展至多台服务器运行，以提供分布式的靶机支撑能力）

在架构设计上，FTC 借鉴了开源项目 A1CTF 的诸多优秀理念和实践，例如清晰的模块划分、合理的接口设计、高效的性能优化以及完善的容器隔离方案等。需要注意的是，为保证系统的简洁轻量，FTC 并未引入 A1CTF 中使用的 Kubernetes 等复杂组件，仅保留了相应的接口以备将来扩展之需。整个 FTC 平台基于 Python 实现，并通过合理的优化手段在可接受范围内满足了高并发场景下的性能需求。

### 5.3 前端架构

前端采用基于 Vue.js 的单页应用（SPA），以组件化方式实现各功能页面，例如首页公告栏、题目列表页面、题目详情弹窗、提交记录/提示组件、排行榜页面、用户资料页面、管理员后台页面等。前端通过 Axios 等工具与后端提供的 REST API 通信，获取或提交 JSON 格式的数据，实现动态交互效果。对于需要实时更新的信息（如排行榜），前端使用 Ajax 定时轮询或结合后端 Django Channels 的 WebSocket 推送来提升实时性。前端界面设计注重简洁友好，突出比赛的关键信息（如倒计时、个人成绩、公告提示等），确保选手操作体验流畅直观。

### 5.4 后端架构

后端采用 Django 框架进行开发，实现按应用模块划分代码结构，例如用户认证模块、比赛管理模块、题目模块、提交与计分模块等。通过 Django ORM 进行数据库读写操作，保证数据的一致性。后端以 RESTful 风格提供 API 接口，向前端返回 JSON 数据。核心服务逻辑包括：用户权限验证、题目 CRUD 及查询接口、Flag 判定流程、分数计算与排名更新、团队相关操作，以及管理员专用的管理功能接口（如题目增删、比赛启动和停止等）。

对于靶机环境的管理，后端采用独立的“靶机管理守护进程”与 Web 主服务并行运行：当 Web 服务收到启动靶机的请求时，将该请求记录到数据库的任务队列中；靶机管理进程持续轮询任务队列并调用 Docker API 启动相应的容器。两者通过数据库共享状态或本机 HTTP 接口通信，实现解耦运行，使 Web 主服务保持无状态，便于水平扩展多个实例。

当前版本中，靶机管理进程与 Web 主服务运行在同一台服务器上；系统架构已预留多节点扩展接口，未来可将靶机管理进程部署到不同服务器节点，通过安全隧道（如 VPN 或 WireGuard）实现跨节点通信。

### 5.5 核心模块设计

- **用户/团队模块**：该模块负责用户与团队相关功能，包括注册、登录、注销、密码修改等操作。FTC 采用 JWT实现认证与会话管理，用户登录后获得加密 Token，用于访问受限接口。系统基于 Django Auth 模块扩展，在用户模型中增加昵称（nickname）、所属战队（team）等字段。团队信息由独立的 Team 模型维护，记录队名、队长、加入令牌等属性，并通过外键关联用户。该模块实现了团队的创建、加入（验证令牌）、退出等逻辑，并限制每位用户在同一比赛中只能加入一支队伍。管理员与普通用户通过角色区分，后台接口统一加上权限验证装饰器（例如 @admin_required）控制访问。注册和登录环节结合 CAP 人机验证，有效防止恶意注册与爆破行为。
- **题目模块**：该模块负责题目的内容管理与复用。Challenge 模型记录题目标题、描述、题型、分值、难度、静态/动态 Flag及附件路径等信息；若题目包含多个子任务，则每个子任务由一个 ChallengeTask 子模型表示。题目标签通过多对多关系关联 Tag 表实现分类筛选。附件文件由管理员上传，系统自动保存并支持 Markdown 格式描述。

FTC 支持两种 Flag 生成方式：静态 Flag 由管理员在后台直接填写；动态 Flag 由系统在选手启动靶机时自动生成，管理员需提供 Flag 模板或参数，系统会为每位选手生成独立 Flag 并注入容器环境中。

题目可导入导出为 JSON 或 ZIP（含JSON与附件），系统导入时自动识别格式、解压并恢复题目信息与附件，但不包含镜像文件。容器环境配置（镜像名或 Compose 模板路径）会随导出文件保存，管理员在导入后只需确保对应镜像已存在即可。

- **竞赛与计分模块**：Contest 模型管理比赛生命周期与计分规则，记录比赛名称、时间、赛制（个人/团队）和封榜时间等信息。比赛与题目为多对多关系，支持一题多赛。Submission 模型记录每次 Flag 提交，包括结果、时间、来源 IP 等，正确提交生成 Solve 记录，错误提交计入尝试次数。FTC 支持 动态分值 与 首杀奖励（First Blood）：题目分值随解出人数自动衰减，首解者获得额外加分。系统使用缓存与任务队列加速排行榜计算，确保高并发下仍能实时更新。比赛结束后可导出成绩与记录，用于复盘与分析。
- **判题模块**：该模块负责验证选手提交的 Flag 是否正确。系统会根据题目配置选择不同的校验方式：若采用默认比对模式，则直接在平台后端比对选手提交内容与系统记录的 Flag；若题目启用了校验脚本，则在受控沙箱中执行脚本完成验证。需要注意的是，静态比对与脚本校验两种模式均适用于静态和动态 Flag，区别在于前者在平台后端完成，后者主要用于靶机侧的复杂验证场景。判题结果会即时反馈至前端，并触发积分更新、排行榜刷新和日志记录。系统为脚本执行设置了超时与资源限制，防止恶意脚本导致性能下降。错误提交会被记录用于赛后分析，同时判题过程产生的日志与状态信息会被保存，以便审计与回溯。
- **靶机管理模块**：该模块负责题目环境的启动、运行与销毁。系统根据题目配置使用 Docker 镜像或 Docker Compose 模板创建容器，并在启动时为动态Flag题目生成独立 Flag。容器的生命周期由平台后台统一管理，系统会记录容器的ID、所属用户、题目、运行状态、端口映射及销毁时间等信息。容器到期或比赛结束后将自动清理，回收系统资源。靶机管理进程与 Web 主服务分离运行，前者负责容器控制，后者负责任务调度，实现逻辑解耦。当前版本采用单机 Docker 运行环境，但架构上已预留多节点接口，未来可通过任务队列或接口扩展，实现分布式靶机管理以支持更大规模赛事。
- **安全模块**：安全机制贯穿 FTC 平台的各个层面。系统启用 Django 安全中间件（防止XSS、CSRF等常见Web攻击），并通过 HTTPS 保障数据传输安全。注册、登录和密码重置环节集成人机验证机制，防止批量注册与暴力破解。平台对用户输入进行严格校验与转义，敏感信息采用加盐哈希存储。容器环境以非特权身份运行并限制系统权限与网络访问，避免题目环境间的相互影响。动态 Flag 由加密随机算法生成，确保唯一性与不可预测性。管理员操作自动记录日志，系统对频繁提交或异常行为设有限速与封禁策略，保证比赛过程安全与公平。

### 5.6 数据模型概述

FTC 采用关系型数据库来存储核心数据，并建立了主要实体及其关联关系。整体数据模型注重保持参照完整性和查询性能。例如，用户、团队、题目、提交记录、比赛等关键实体均设计为独立的数据表，通过外键建立实体之间的关联，并对高频查询的字段添加索引以优化性能。尤其是题目与比赛采用多对多关系，实现了“题库中心”的题目复用模式。在数据表中也适当增加了一些辅助字段，以满足功能扩展或安全审计的需要（例如在提交记录中添加提交 IP 地址、首解标志等字段）。所有涉及成绩变动的操作都通过事务处理确保原子性，避免并发情况下数据不一致。

### 5.7 数据库设计

主要的数据表设计和关系如下：

- **用户表（Users）**：存储用户账户的基本信息（用户名、密码哈希、邮箱等）以及用户属性（如昵称、所属战队等）。通过外键与团队表关联，表示用户所属的战队。
- **团队表（Teams）**：存储战队信息（队名、队长、加入令牌等）。通过外键关联队长用户，并通过中间关联表关联队员列表，实现用户与团队的多对一关系（每位用户至多属于一个团队）。
- **题目表（Challenges）**：存储题目的基本信息（题目标题、描述、类别、初始分值、题目类型如动态/静态、标准 Flag 或校验脚本路径等）。通过多对多关系关联比赛表，以支持题目在不同比赛中的复用；同时通过多对多关系关联标签表（Tags），以支持题目的分类筛选。
- **子任务表（ChallengeTasks，可选）**：用于表示包含多个子 Flag 的复杂题目。每个子任务记录一个 Flag 及其分值，外键关联到所属的主题目（Challenge）。若不需要多子任务，此表可以不使用。
- **提交记录表（Submissions）**：记录选手每一次提交 Flag 的情况，包括提交者（用户或队伍）、关联的题目或子任务、提交时间、提交内容（Flag 文本）、判定结果（正确/错误）等信息。该表还可以包含提交来源 IP、是否首解等字段，用于安全审计和特殊计分规则。
- **比赛表（Contests）**：存储比赛的配置与状态信息（比赛名称、开始时间、结束时间、赛制类型、封榜时间等）。通过比赛-题目关联表与 Challenges 表形成多对多关系，以实现题目复用和赛题管理。
- **公告表（Announcements）**：存储比赛公告信息（公告内容、发布时间等），通过外键关联所属比赛。

数据库设计中，为所有关键的外键都建立了索引，并严格设置了关联删除/更新策略以保证引用完整性。例如，删除比赛时可以选择同时删除关联的公告记录、解绑相关题目等（视具体业务规则决定）。题目与比赛的多对多关联通过中间表实现，使得同一道题目可以被多个比赛引用，从而支持题库中心功能。Submission 提交表中增加的提交 IP、首解标记等字段方便进行安全监控和实现特殊计分策略。所有涉及得分变动的数据库更新操作都通过事务（Transaction）处理，确保加减分操作的原子性，防止并发提交造成计分错误。

### 5.8 部署与扩展

**部署方案**：FTC 支持多种部署模式。平台提供了基于 Docker Compose 的一键部署方案，将 Django 后端、MySQL 数据库、Redis 缓存、前端静态资源服务器等封装为独立容器，通过编排脚本一并启动。其中 Django 容器包含应用代码并通过 Gunicorn 提供 Web 服务，Nginx 容器作为前端的反向代理，MySQL 和 Redis 容器则分别提供数据存储和缓存服务。采用 Docker 容器化部署能够让 FTC 在任何支持 Docker 的主机上方便运行，并可通过环境变量配置数据库、缓存等参数，提高部署的可移植性和灵活性。

当需要支持更大规模的赛事（数百乃至上千人参赛）时，可以按照功能模块对 FTC 进行拆分部署，对各组件进行扩展和横向扩容：

- **数据库扩展**：将 MySQL 数据库升级为主从架构，或采用分布式数据库方案，以提高读取性能和可靠性。缓存服务器 Redis 则可以启用主从复制或集群模式，避免单点故障造成性能瓶颈。
- **后端扩展**：由于后端服务本身无状态，可在前端增加负载均衡组件（如 Nginx 或 HAProxy），并启动多个 Django 后端实例来分担请求压力。同时配合使用缓存和只读从库来减轻主数据库的负载。
- **靶机扩展**：增加多台靶机服务器节点并安装 Docker，引入额外的靶机管理守护进程，使容器可以分布式地在不同主机上创建。通过任务队列或题目中心的统一调度接口，实现跨节点的容器启动，从而支撑大规模并发的靶机环境请求。FTC 的容器调度模块与主站服务解耦设计，这意味着即使需要将靶机管理扩展到 Kubernetes 等容器编排系统，仅需对接预留的接口即可实现，无需修改核心代码逻辑。
- **功能扩展接口**：FTC 在架构上预留了插件机制，对于特定的新需求可以通过加载插件模块来实现。比如，支持新的题目类型（如动态生成题目、解谜题等）时，可以通过实现预定义接口将其挂载到题目系统中。又如，需要替换默认的用户认证方式，FTC 提供 OAuth2 对接接口，开发对应的插件实现 OAuth2 登录逻辑即可无缝替换本地用户名密码登录。同理，各种第三方服务的集成（如邮件通知、云存储附件等）也可以通过配置开关和模块注入的方式进行扩展。
- **预留扩展支持**：FTC 当前实现了 Jeopardy 模式（包括团队赛和个人赛），全面支持传统CTF比赛，且在架构上预留了对不同赛制的扩展可能性。例如，如需支持 Attack-Defense（攻防对抗）赛制，选手需要互相攻击对方的主机并提交战报。这种情况下可以利用 FTC 模块化的设计，新增“战队主机分配”模块、实时攻击计分模块等，而不破坏已有的 Jeopardy 功能。FTC 的计分体系通过“得分项”而非将固定题目分数直接绑定到用户，因此可以灵活地支持手动加减分或来自外部系统的积分，为将来拓展支持攻防赛等复杂赛制奠定了基础。

**配置与维护**：为降低运维成本，FTC 提供了统一的配置文件来管理关键参数（如数据库连接信息、缓存地址、第三方 API 密钥等），运维人员能够集中修改主要配置项。FTC 提供详尽的安装部署文档和运维手册，指导准备运行环境（如 Python 依赖安装、Nginx 配置示例等）。在平台运行过程中，管理员可以通过后台界面或运维脚本查看 FTC 的实时状态（如当前在线用户数、运行中的靶机容器列表等），并具备一定程度的自我诊断能力。完善的日志机制会将错误堆栈等调试信息记录到日志文件中，便于开发者和运维人员排查问题。
