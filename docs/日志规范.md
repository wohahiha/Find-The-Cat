# FTC 日志规范 / 标准

面向对象：所有后端开发、运维、测试。目标是让日志“字段统一、位置可预期、便于检索/导出/审计”。

## 1. 日志总体设计
- 日志入口：统一使用 `apps.common.infra.logger.get_logger` 获取 logger，不允许直接 `print`。
- 配置来源：`Config/settings.py` 提供默认值，后台 **系统配置**（`SystemConfig`）可覆盖 `LOG_PATH`，修改后重启生效。
- 输出位置：默认写入 `logs/system.log`，按日轮转，保留 30 天；轮转后文件名如 `system.log.2025-12-02`。
- 可读格式：默认 PLAIN，兼容 JSON。格式化器在 `apps/common/infra/logger.py`；格式互转工具在 `apps/common/utils/log_formatter.py`。
- 上下文注入：`RequestContextMiddleware` + `request_context` 提供 `request_id/user_id/account_id/username/ip/path/method/user_agent`，自动注入到日志中（JSON 格式包含 request_id，PLAIN 以上下文尾注形式）。
- 审计出口：后台 Django Admin 的系统日志页依赖 `apps.system.log_models.LogParser` 解析文件，可导出 TXT/CSV/JSON。

## 2. 配置与开关
- 日志级别：默认 `INFO`，当前固定在 `apps.common.infra.logger.configure_logging` 中。如需调整级别，请在部署前显式修改 `configure_logging(level=...)` 或自定义 settings；后台暂不提供动态调整。
- `LOG_PATH` 日志目录，默认 `logs/`。目录不存在会自动创建；无写权限会导致启动报错。
- 调试模式：`DEBUG=True` 且环境变量 `DEBUG=true` 时会额外输出到控制台（开发场景）。
- 配置加载顺序：**后台 SystemConfig > settings.py 默认值**。启动早期会先用 settings 值配置 logger，数据库就绪后由后台覆盖。

## 3. 输出格式
### PLAIN（默认，便于人读和 grep）
```
2025-12-02 10:15:30 INFO apps.contests.services 创建比赛 [wohahiha|1|127.0.0.1|/api/contests/]
```
结构：`{timestamp} {level} {logger} {message} [username|account_id|ip|request_path]`。异常会追加换行+堆栈。

### JSON（便于机器采集/审计）
```
{"timestamp":"2025-12-02 10:15:30","level":"INFO","logger":"apps.machines.services","message":"靶机启动成功","username":"alice","user_id":3,"account_id":1001,"ip_address":"127.0.0.1","request_path":"/api/machines/start/","request_id":"abc123def456"}
```
字段同下表，异常时包含 `exception`。

## 4. 标准字段
- `timestamp`：`YYYY-MM-DD HH:MM:SS`，服务端时间。
- `level`：`INFO/WARNING/ERROR/CRITICAL`。
- `logger`：模块路径，推荐 `__name__`，用于来源归类（accounts/contests/challenges/submissions/machines/problem_bank/system/django）。
- `message`：简短中文事件描述。
- `username/user_id/account_id`：来自请求上下文；未登录则为空。
- `ip_address`：客户端 IP，取 `X-Forwarded-For` 首段或 `REMOTE_ADDR`。
- `request_path`：请求路径；后台任务为空。
- `request_id`：仅 JSON 输出，来自 `RequestContextMiddleware`（或 `generate_request_id`）。用于跨服务链路追踪。
- 业务自定义字段：通过 `extra=logger_extra({...})` 注入，敏感键（password/token/flag/code/email_code/auth_code/captcha 等）会被 `***` 脱敏。

## 5. 采集与轮转
- Handler：`TimedRotatingFileHandler` 每日午夜轮转，保留 30 天。
- 文件编码：UTF-8。
- 线程安全：根 logger 统一配置；重复调用 `configure_logging` 会清理旧 handler，避免重复写入。
- Windows 兼容：自定义 `SafeTimedRotatingFileHandler` 处理轮转时的 `PermissionError`。

## 6. 使用规范（开发侧）
1) 获取 logger：`logger = get_logger(__name__)`。  
2) 写日志：`logger.info("创建比赛", extra=logger_extra({"contest": contest.slug}))`。  
3) 级别约定：  
   - INFO：正常业务事件（登录成功、创建/导入/导出、靶机启动/停止、判题结果等）。  
   - WARNING：可恢复异常或退路（缓存降级、重复操作、端口占用回退等）。  
   - ERROR/CRITICAL：不可恢复、需要告警的异常（数据库/Redis/Docker 连接失败、判题崩溃等）。  
4) 禁止直接输出敏感数据；如需记录参数，统一通过 `logger_extra`，会自动脱敏。  
5) 避免在库/模型层调用 logger_extra 以外的 raw extra，避免敏感信息泄漏。  
6) 请求外场景（Celery/管理命令）也可以直接用 `get_logger`，上下文字段为空但格式一致。  

## 7. 请求上下文
- 中间件：`apps.common.middleware.RequestContextMiddleware` 在请求进入时写入 `request_id/user/路径/IP/UA`，响应或异常时清理，最后一次上下文保留 2 秒以供日志尾部使用。
- JWT 鉴权：`apps.common.authentication.update_request_user` 在认证完成后更新用户上下文（中间件运行时用户未加载）。
- 依赖：`apps/common/utils/request_context.py` 维护上下文 `contextvars`，日志格式化器在输出时读取。

## 8. 系统审计与导出
- 日志解析：`apps.system.log_models.LogParser` 自动识别 PLAIN/JSON，统一为 `LogEntry`，并按 logger 分类（业务/系统/框架）。未知 logger 归类为 system。
- 后台入口：Django Admin 的系统日志列表，可筛选、分页、搜索关键字。
- 导出：支持 TXT/CSV/JSON（三个 admin 动作），文件名包含导出时间戳。
- 日志类别视图：`apps.system.models.SystemLog/SystemLogCategory` 映射到虚拟表用于 Admin 展示。

## 9. 关键业务的结构化日志（现状）
- 判题：`apps.submissions.services` 记录正确/错误提交，附带 `challenge`, `contest`, `user_id`, `flag_length`, `hint_cost`, `awarded_points` 等。
- 靶机：`apps.machines.services` 与 `apps.machines.tasks` 记录端口分配/释放、容器启动/销毁、TTL/redis_key、异常堆栈。
- 题库导入导出：`apps.problem_bank.services` 记录导入/导出数量、关联比赛、题目 slug。
- 比赛与队伍：`apps.contests.services` 记录创建比赛/公告、队伍创建/加入/转移/解散，以及管理员操作。
- 系统配置：`apps.system.admin` 与 `apps.system.services` 记录配置变更与审计导出。

## 10. 操作指引
- 查看实时日志：`tail -f logs/system.log`（生产注意权限与 logrotate）。轮转文件附日期后缀。
- 调整级别/路径：后台“系统配置”可修改 `LOG_PATH` 后重启；如需调整日志级别，请在部署前修改 `apps.common.infra.logger.configure_logging` 或 settings。
- 新增日志时：
  1. 在 Service/任务/视图中获取 logger；
  2. 确保消息为中文，描述清晰；
  3. 使用 `logger_extra` 附加业务关键字段（ID/slug/数量/TTL 等），避免敏感数据；
  4. 选择合适级别（见上）。
- 故障排查：若日志缺失，检查 `LOG_PATH` 权限、磁盘空间、日志级别是否过高，以及中间件是否在 `MIDDLEWARE` 中。

## 11. 未来扩展建议
- 引入集中式采集：如接入 ELK/Vector/FluentBit，可直接解析现有 PLAIN/JSON，建议优先用 JSON。
- 增加告警：结合 ERROR/CRITICAL 级别对接告警渠道；使用 `request_id` 进行跨服务链路关联。
- 后台 UI：可按 `request_id/user_id/contest/challenge` 增加过滤器，提升审计效率。
