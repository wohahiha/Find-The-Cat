# FTC 系统日志标准

## 一、概述

本文档定义了 FTC（Find The Cat）CTF 平台的统一日志标准，所有模块的日志记录必须遵循此标准。

**设计原则**：
- **统一性**：所有日志遵循统一的格式和字段标准
- **可解析性**：日志易于程序解析和人类阅读
- **可扩展性**：支持未来新增字段和格式
- **低耦合性**：日志模块独立于业务模块，能适应业务调整

---

## 二、日志字段标准

### 2.1 必需字段

每条日志必须包含以下核心字段：

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| **timestamp** | datetime | 日志记录时间（精确到秒） | 2025-11-28 16:57:25 |
| **level** | string | 日志级别（重要性） | INFO, WARNING, ERROR, CRITICAL |
| **logger** | string | Logger名称（模块路径） | apps.accounts.services |
| **message** | string | 日志消息内容 | 用户登录成功 |

### 2.2 可选字段（上下文信息）

以下字段在有上下文信息时应该包含：

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| **username** | string | 操作用户的用户名 | admin |
| **user_id** | integer | 操作用户的数据库ID | 1 |
| **account_id** | integer | 操作用户的账户ID | 1 |
| **ip_address** | string | 客户端IP地址 | 127.0.0.1 |
| **request_path** | string | HTTP请求路径 | /api/accounts/auth/login/ |
| **request_id** | string | 请求追踪ID（UUID） | a1b2c3d4-e5f6-... |

### 2.3 扩展字段

业务模块可以根据需要添加额外字段，但必须确保不与标准字段冲突。

## 三、日志级别定义

### 3.1 级别说明

FTC 系统使用以下四个日志级别：

| 级别 | 说明 | 使用场景 |
|------|------|----------|
| **INFO** | 常规信息 | 正常业务操作（登录、创建、查询等） |
| **WARNING** | 警告信息 | 潜在问题但不影响运行（验证失败、资源不足等） |
| **ERROR** | 错误信息 | 程序错误但可恢复（API调用失败、数据验证错误等） |
| **CRITICAL** | 严重错误 | 系统级错误需要立即处理（数据库连接失败、服务崩溃等） |

### 3.2 与日志输出级别的区别

- **日志级别（level）**：表示某条日志的重要性
- **日志输出级别（LOG_OUTPUT_LEVEL）**：系统配置，控制哪些级别的日志会被输出到文件

**LOG_OUTPUT_LEVEL 可选值**：

| 输出级别 | 说明 | 包含的日志级别 | 使用场景 |
|---------|------|----------------|----------|
| **Low** | 仅错误 | ERROR、CRITICAL | 生产环境，仅关注错误 |
| **Medium** | 包含警告 | WARNING、ERROR、CRITICAL | 推荐用于生产环境 |
| **High** | 全部日志 | INFO、WARNING、ERROR、CRITICAL | 开发环境或问题排查 |

示例：如果 `LOG_OUTPUT_LEVEL = Medium`，则只有 WARNING、ERROR、CRITICAL 级别的日志会被写入文件。

## 四、日志分类标准

### 4.1 类型分类（大类）

根据 logger 名称自动判断日志类型：

| 类型 | 判断规则 | 说明 |
|------|----------|------|
| **业务日志** | logger.startswith('apps.') | 应用业务逻辑日志 |
| **框架日志** | logger.startswith('django.') | Django框架日志 |
| **系统日志** | logger in ['system', 'celery'] | 系统级别日志 |
| **其他** | 不匹配以上规则 | 第三方库或其他日志 |

### 4.2 来源分类（小类）

从 logger 名称中提取具体模块名：

- `apps.accounts.services` → 来源：`accounts.services`
- `apps.system.apps` → 来源：`system.apps`
- `django.request` → 来源：`django.request`

**提取规则**：
- 业务日志：去除 `apps.` 前缀
- 其他日志：保持原始 logger 名称

## 五、日志格式标准

### 5.1 支持的格式

FTC 日志系统支持两种格式：

#### A. JSON 格式

**特点**：
- 结构化存储，易于程序解析
- 包含完整的字段信息
- 适用于：业务日志、管理员操作日志

**格式定义**：
每行一个 JSON 对象，包含所有标准字段。

**示例**：
```json
{"timestamp": "2025-11-28 16:57:25", "level": "INFO", "logger": "apps.accounts.services", "message": "用户登录成功", "username": "admin", "account_id": 1, "ip_address": "127.0.0.1", "request_path": "/api/accounts/auth/login/"}
```

#### B. PLAIN 格式

**特点**：
- 纯文本格式，人类可读
- 固定格式，便于 grep 等工具处理
- 适用于：框架日志、系统启动日志

**格式定义**：
```
{timestamp} {level} {logger} {message} [{username}|{account_id}|{ip_address}|{request_path}]
```

**示例**：
```
2025-11-28 16:57:25 INFO apps.accounts.services 用户登录成功 [admin|1|127.0.0.1|/api/accounts/auth/login/]
```

**上下文字段为空时**：
```
2025-11-28 16:57:25 INFO django.utils.autoreload Watching for file changes [-|-|-|-]
```

### 5.2 格式转换

所有日志必须支持在 JSON 和 PLAIN 格式之间互相转换。

**转换函数**：
- `to_json(log_dict) -> str`：将日志字典转换为 JSON 字符串
- `to_plain(log_dict) -> str`：将日志字典转换为 PLAIN 字符串
- `parse_json(json_str) -> dict`：解析 JSON 格式日志
- `parse_plain(plain_str) -> dict`：解析 PLAIN 格式日志

**转换模块位置**：`apps/common/utils/log_formatter.py`

## 六、日志文件组织标准

### 6.1 目录结构

```
logs/
├── system.log              # 当前主日志文件
├── system.log.2025-11-28   # 按日期轮转的历史日志
├── system.log.2025-11-27
└── archive/                # 归档目录（可选）
    └── system.log.2025-11.gz
```

### 6.2 文件命名规则

- **主日志文件**：固定名称 `system.log`
- **轮转文件**：`system.log.{YYYY-MM-DD}` 或 `system.log.{number}`
- **归档文件**：可压缩为 `.gz` 格式

### 6.3 日志轮转策略

**按日期轮转**（推荐）：
- 每天午夜创建新文件
- 旧文件自动重命名为 `system.log.{date}`
- 使用 `TimedRotatingFileHandler`

**按大小轮转**：
- 超过指定大小（如 100MB）创建新文件
- 使用 `RotatingFileHandler`

### 6.4 日志保留策略

- 保留最近 30 天的日志文件
- 超过 30 天的日志可以归档或删除
- 重要日志（ERROR/CRITICAL）可以单独归档

## 七、日志记录标准接口

### 7.1 Logger 获取

所有模块必须使用以下方式获取 logger：

```python
import logging
logger = logging.getLogger(__name__)  # 自动获取模块路径，禁止硬编码
```

**禁止使用**：
```python
logger = logging.getLogger('system')  # ❌ 硬编码名称
logger = logging.getLogger('my_logger')  # ❌ 自定义名称
```

### 7.2 日志记录方法

```python
# INFO 级别
logger.info("用户登录成功", extra={
    'user': user,
    'ip': ip,
    'request_path': request.path
})

# WARNING 级别
logger.warning("验证码已过期", extra={
    'email': email,
    'ip': ip
})

# ERROR 级别
logger.error("数据库查询失败", exc_info=True, extra={
    'query': query_str,
    'user': user
})

# CRITICAL 级别
logger.critical("Redis 连接失败", exc_info=True)
```

### 7.3 自动上下文注入

日志记录器应该自动从请求上下文中获取以下信息：
- 当前用户（user）
- 客户端IP（ip_address）
- 请求路径（request_path）
- 请求ID（request_id）

业务代码无需手动传递这些参数（除非需要覆盖）。

## 八、日志解析标准

### 8.1 LogEntry 类

表示一条解析后的日志，提供统一的访问接口。

**核心属性**：
- `id`：日志行号
- `timestamp`：datetime 对象
- `level`：日志级别
- `logger_name`：logger 名称
- `message`：日志消息
- `username`、`user_id`、`account_id`：用户信息
- `ip_address`、`request_path`：请求信息

**核心方法**：
- `get_type()`：返回日志类型（业务日志/框架日志/系统日志/其他）
- `get_source()`：返回日志来源（提取模块名）
- `get_summary(max_length)`：返回消息摘要（智能截断）
- `to_json()`：转换为 JSON 格式字典
- `to_plain()`：转换为 PLAIN 格式字符串

### 8.2 LogParser 类

解析日志文件，支持 JSON 和 PLAIN 两种格式。

**核心方法**：
- `parse_file(file_path, filters)`：生成器，逐行解析文件，返回 LogEntry 对象
- `parse_line(line, line_number)`：解析单行日志，返回 LogEntry 对象或 None
- `detect_format(line)`：检测日志行的格式（JSON 或 PLAIN）
- `parse_json_line(line, line_number)`：解析 JSON 格式日志
- `parse_plain_line(line, line_number)`：解析 PLAIN 格式日志

**边解析边过滤**：
- 支持在解析时应用过滤条件
- 只解析符合条件的日志行
- 提高性能，减少内存占用

## 九、系统配置

### 9.1 日志相关配置项

| 配置项 | 说明 | 类型 | 可选值 | 默认值 |
|--------|------|------|--------|--------|
| **LOG_OUTPUT_LEVEL** | 日志输出级别（控制哪些日志会被写入文件） | 选择 | Low / Medium / High | Medium |
| **LOG_PATH** | 日志目录路径 | 字符串 | - | logs/ |

**LOG_OUTPUT_LEVEL 详细说明**：
- **Low**：仅输出 ERROR、CRITICAL（适合生产环境，仅关注错误）
- **Medium**：输出 WARNING、ERROR、CRITICAL（推荐用于生产环境）
- **High**：输出 INFO、WARNING、ERROR、CRITICAL（适合开发环境或问题排查）

### 9.2 已移除的配置项

- ~~`LOG_LEVEL`~~（重命名为 `LOG_OUTPUT_LEVEL`，并改用 Low/Medium/High 级别）
- ~~`LOG_FORMAT`~~（由日志标准统一规定，无需配置）
- ~~`LOG_FILE`~~（重命名为 `LOG_PATH`，只指定目录）

## 十、兼容性和扩展性

### 10.1 向后兼容

- 日志解析器必须能够处理旧格式的日志
- 对于无法解析的日志行，尽量提取基本信息（时间、级别、消息）
- 不因旧日志格式导致系统崩溃

### 10.2 格式混合

- 同一日志文件中可以混合 JSON 和 PLAIN 格式
- 解析器自动识别每行的格式
- 统一转换为 LogEntry 对象

### 10.3 扩展新字段

- 业务模块可以添加自定义字段
- 自定义字段通过 `extra` 参数传递
- 确保不与标准字段冲突

## 十一、最佳实践

### 11.1 日志记录

✅ **推荐做法**：
- 使用 `__name__` 获取 logger
- 包含必要的上下文信息
- 选择合适的日志级别
- 消息简洁明了，描述操作结果
- ERROR 级别包含异常信息（`exc_info=True`）

❌ **不推荐做法**：
- 硬编码 logger 名称
- 日志消息过长（应该截断或只记录关键信息）
- 在循环中记录大量日志（影响性能）
- 记录敏感信息（密码、token 等）

### 11.2 日志查看

- 使用后台"系统日志"页面查看和过滤
- 支持按级别、类型、来源、账户类型过滤
- 支持导出为 CSV/JSON/TXT 格式

### 11.3 日志分析

- ERROR/CRITICAL 日志需要及时处理
- 定期检查 WARNING 日志，发现潜在问题
- 利用日志分析用户行为和系统性能

### 11.4 输出级别建议

- **生产环境**：使用 Medium 级别，平衡日志量和信息完整性
- **开发环境**：使用 High 级别，获取完整的执行信息
- **问题排查**：临时切换到 High 级别，排查完成后恢复 Medium
- **低负载场景**：如果磁盘空间充足，可以始终使用 High 级别保留更多信息
