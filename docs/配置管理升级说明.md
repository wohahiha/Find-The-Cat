# FTC 配置管理升级说明

## 概述

**已完成改造：去掉 .env 依赖，改为"后台配置 + settings.py 默认值"的管理模式**

## 改造内容

### 1. 去掉 .env 文件依赖

- **之前**：配置通过 .env 文件管理，需要 SSH 到服务器修改
- **现在**：所有配置默认值在 `Config/settings.py` 中硬编码，管理员通过 Django Admin 后台"基础配置"可视化修改

### 2. 配置优先级

```
┌─────────────────────────────────────┐
│ 优先级 1：后台配置（SystemConfig 表）│  ← 最高优先级
├─────────────────────────────────────┤
│ 优先级 2：settings.py 硬编码值       │  ← 默认值
├─────────────────────────────────────┤
│ 优先级 3：代码传入的 default 参数    │  ← 兜底值
└─────────────────────────────────────┘
```

### 3. 自动初始化机制

Django 启动时（`apps.system.apps.SystemConfig.ready()`）会自动：
- 读取 `SUPPORTED_CONFIGS` 中定义的 33 个配置项
- 从 `settings.py` 读取默认值
- 写入数据库 `SystemConfig` 表（仅首次或缺失时创建）
- 已存在的配置不会被覆盖（保留管理员修改的值）

### 4. 配置分类

#### 启动必需配置（修改后需重启服务）
- `SECRET_KEY`：Django 安全密钥
- `DEBUG`：调试模式开关
- `ALLOWED_HOSTS`：允许访问的主机列表
- `DB_ENGINE`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`：数据库配置
- `REDIS_HOST`, `REDIS_PORT`, `REDIS_DB_CACHE`：Redis 缓存配置
- `CELERY_BROKER_URL`, `CELERY_RESULT_BACKEND`：Celery 配置

#### 运行时配置（部分修改后立即生效）
- `LOG_LEVEL`, `LOG_FORMAT`, `LOG_FILE`：日志配置
- `DOCKER_*`：Docker 靶机相关配置
- `MACHINE_*`：靶机运行时参数
- `SCOREBOARD_CACHE_TTL`：记分板缓存时间

## 首次部署流程

### 1. 克隆代码

```bash
git clone <repository_url>
cd FTC
```

### 2. 安装依赖

```bash
# 激活虚拟环境
FTCVenv\Scripts\activate  # Windows
source FTCVenv/bin/activate  # Linux/Mac

# 安装依赖
pip install -r requirements.txt
```

### 3. 数据库迁移

```bash
python manage.py migrate
```

### 4. 创建超级管理员

```bash
python manage.py createsuperuser
```

### 5. 启动服务

```bash
python manage.py runserver
```

**注意**：首次启动时，Django 会自动将 `settings.py` 的默认值填入后台"基础配置"。

### 6. 登录后台修改敏感配置

访问 http://localhost:8000/admin 登录后台：

1. 进入 **SYSTEM** → **基础配置**
2. 必须修改的配置（生产环境）：
   - **SECRET_KEY**：改为长度 50+ 的强随机字符串
   - **DEBUG**：改为 `False`
   - **ALLOWED_HOSTS**：改为实际域名/IP，如 `["example.com", "api.example.com"]`
   - **ALLOW_LOGIN_WITHOUT_CAPTCHA**：改为 `False`

3. 根据实际环境修改：
   - 数据库配置（如需使用 MySQL/PostgreSQL）
   - Redis 配置（如使用远程 Redis）
   - Docker 配置（如使用远程 Docker 节点）
   - 日志配置（如调整日志级别）

### 7. 重启服务

修改启动必需配置后，必须重启服务才能生效：

```bash
# Ctrl+C 停止服务，然后重新启动
python manage.py runserver

# 生产环境使用 Gunicorn
gunicorn Config.wsgi:application --bind 0.0.0.0:8000
```

## 生产环境部署建议

### 1. SECRET_KEY 安全性

- 使用强随机字符串，长度 50+ 字符
- 包含大小写字母、数字、特殊字符
- 不要复用其他环境的 SECRET_KEY
- 示例生成方式：
  ```python
  python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"
  ```

### 2. 数据库切换

切换到 MySQL/PostgreSQL：
1. 在后台"基础配置"修改 `DB_ENGINE` 为 `mysql` 或 `postgres`
2. 填写 `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`
3. 重启服务
4. 运行迁移：`python manage.py migrate`

### 3. Redis 配置

- 确保 Redis 服务已启动
- 修改 `REDIS_HOST`, `REDIS_PORT` 指向实际 Redis 服务器
- 建议使用独立的 DB 索引避免冲突：
  - `REDIS_DB_CACHE`：0（缓存、限流）
  - `REDIS_DB_CELERY`：1（Celery Broker）
  - `REDIS_DB_CELERY_RESULT`：2（Celery Result）

### 4. Celery 异步任务

启动 Celery Worker 和 Beat：

```bash
# Worker（处理异步任务）
celery -A Config worker -l info

# Beat（定时任务调度器，如靶机清理）
celery -A Config beat -l info

# Windows 环境
celery -A Config worker --pool=solo
```

### 5. 邮件配置

邮件配置在 **accounts** 模块的 **发信账号** 中单独管理：
1. 进入 **Django 管理** → **ACCOUNTS** → **发信账号**
2. 添加 SMTP 账号（QQ/163/Gmail/Outlook 或自定义）
3. 设置默认账号

## 配置查看与修改

### 后台界面

访问 http://localhost:8000/admin → **SYSTEM** → **基础配置**

每个配置项显示：
- **键**：配置项名称（只读）
- **值类型**：字符串/整数/布尔/JSON/密钥（只读）
- **配置值**：当前值（敏感字段显示为 `******`）
- **值预览**：实际生效的值（后台配置 > settings.py）
- **说明**：配置项简短描述
- **详细用途说明**：配置项详细说明和注意事项
- **是否必填**：标记必填配置
- **敏感字段**：标记敏感配置（如密码）

### 编程方式

```python
from apps.system.services import ConfigService

cs = ConfigService()

# 读取配置
secret_key = cs.get('SECRET_KEY')
redis_host = cs.get('REDIS_HOST', '127.0.0.1')

# 修改配置后清理缓存
cs.invalidate('SECRET_KEY')
```

## 常见问题

### Q1: 修改配置后为什么没生效？

**A1**:
- **启动必需配置**（SECRET_KEY、数据库、Redis等）需要重启服务
- **运行时配置**（部分）修改后立即生效
- 修改后建议调用 `cs.invalidate(key)` 清理缓存

### Q2: 如何恢复默认配置？

**A2**:
1. 在后台"基础配置"删除对应配置项
2. 重启服务，会自动从 `settings.py` 重新创建

### Q3: Redis 不可用会影响功能吗？

**A3**:
- 配置服务会自动降级，跳过缓存直接查数据库
- 不影响配置读取，仅性能略降
- 建议生产环境保持 Redis 可用

### Q4: 数据库中有旧配置项怎么办？

**A4**:
- `ensure_supported_configs()` 不会删除旧配置
- 手动在后台删除无用配置即可
- 或保留旧配置以便回滚

### Q5: 启动时看到警告"Accessing the database during app initialization"？

**A5**:
- 这是 Django 的提示，不影响功能
- 我们的场景确实需要在启动时初始化配置
- 可以忽略这个警告

## 升级前后对比

| 项目 | 升级前 | 升级后 |
|------|--------|--------|
| 配置文件 | .env 文件 | settings.py + 后台 |
| 修改方式 | SSH 编辑 .env | 后台可视化修改 |
| 默认值来源 | .env 或代码硬编码 | settings.py 硬编码 |
| 配置可见性 | 需要 SSH 查看 | 后台直接查看 |
| 敏感信息 | 明文在 .env 中 | 后台脱敏显示 |
| 版本控制 | .env 不宜提交 | settings.py 可安全提交 |
| 生效方式 | 重启服务 | 部分立即生效，部分需重启 |
| 初始化 | 手动创建 .env | 自动从 settings.py 填充 |

## 注意事项

1. **首次部署后立即修改敏感配置**（SECRET_KEY、DEBUG、ALLOWED_HOSTS）
2. **生产环境必须关闭 DEBUG**，否则会泄露敏感信息
3. **定期备份数据库**，配置数据存储在 SystemConfig 表中
4. **修改启动必需配置后需重启服务**
5. **邮件配置在 accounts 模块单独管理**，不在"基础配置"中

## 技术细节

### 配置初始化时机

```python
# apps/system/apps.py
class ConfigsConfig(AppConfig):
    def ready(self):
        # Django 启动后自动执行
        ConfigService().ensure_supported_configs()
```

### 配置读取优先级

```python
# apps/system/services.py
def get(self, key: str, default: Any = None) -> Any:
    # 1. 尝试从缓存读取
    cached = cache.get(cache_key)

    # 2. 从数据库读取后台配置（优先级最高）
    cfg = self.repo.get_by_key(key)

    # 3. 从 settings.py 读取默认值
    value = getattr(settings, key, None)

    # 4. 使用传入的 default 参数
    return value or default
```

### 缓存降级机制

```python
try:
    cache.get(key)
except Exception as e:
    logger.warning(f"缓存读取失败，降级到数据库查询: {e}")
    # 继续执行，不影响功能
```
